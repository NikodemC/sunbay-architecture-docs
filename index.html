Sunbay — jak Google Sheets są wpięte w architekturę

Dokumentacja dla data engineera wspierającego wyjście z Google Sheets.
Stan na 2026-05-29 · źródło: kod repozytoriów Sunbay (backend .NET 10) i Sunbay-frontend (React) · zakres: żywa ścieżka produkcyjna, arkusze w centrum.
TL;DR

Arkusz Google to nie „głupie wejście" — to warstwa ETL/wzbogacania w środku przepływu. Aktywne dziś procesy windykacyjne (CollectionFlow, RecurringCollectionFlow) są zasilane wyłącznie z Google Sheets — bez wyjątku. InvoicingFlow używa tego samego wzorca, ale jest obecnie nieużywany (patrz „Zakres na dziś").

Pełny cykl życia arkusza:

  • A · Zasilanie — Sunbay sam ciągnie faktury/kontakty z softów księgowych (Xero, Fakturownia, inFakt, SaldeoSmart, wFirma, Systim, Zoho) i zapisuje je do arkusza. Wyzwala to Apps Script w arkuszu wołając endpointy */sheets-export.
  • B · Transformacja — w samym arkuszu (poza kodem): formuły, joiny z innymi źródłami (np. adresy e-mail), wyliczane pola pochodne.
  • C · Import — timer tika co minutę tylko po to, by sprawdzić, czy któryś proces jest zaplanowany na ten moment; każdy proces odświeża arkusz i wysyła najwyżej raz dziennie (o swojej godzinie). Gdy proces jest „due": odczyt arkusza (Sheets API) → import wierszy do bazy.
  • D · Downstream — dalej: windykacja (e-mail/SMS) albo generowanie faktur w systemie docelowym.

Most na dowolne pola policzone w arkuszu: kolumny spoza modelu C# trafiają automatycznie do CustomFields i przepływają do importu bez zmian w kodzie.

Co realnie trzeba odtworzyć przy migracji: nie pojedynczy „import", lecz cały łańcuch pull → enrich → compute, który dziś jest rozmazany między kodem API a formułami żyjącymi w arkuszach.

Zakres na dziś

Aktywne procesy: Collection Flow i Recurring Collection Flow — zasilane z arkuszy, napędzają windykację (e-mail/SMS). Na nich skupiamy się teraz.

Invoicing Flow: zbudowany tym samym wzorcem (arkusz → import), ale obecnie nieużywany i pozostanie taki jeszcze przez jakiś czas. Opisany niżej dla kompletności — ale nie jest dziś w użyciu.

OCR (Gemini / Vertex AI): osobny pipeline; jego wyniki docelowo zasilają arkusze pod Invoicing Flow — czyli dziś również poza realnym użyciem.

Kierunek V2: dane z arkuszy mają zasilać AI-analitykę; prawdopodobnie pojawi się wtedy warstwa pośrednia (semantic layer / DuckDB lub podobne) — to naturalny punkt zaczepienia dla migracji.

1. Architektura w pigułce

Sunbay to platforma automatyzacji fakturowania i windykacji: React SPA + .NET API (Clean Architecture, MediatR CQRS) na bazie MSSQL, z Azure Functions (timery, konsumenci Service Bus, Durable) i Azure Service Bus jako asynchronicznym kręgosłupem. Integruje się z zewnętrznymi softami księgowymi. Google Sheets jest centralnym hubem danych między softami księgowymi a procesami Sunbay.

OCR (Gemini / Vertex AI) celowo nie jest na diagramie — to osobny pipeline ekstrakcji dokumentów, którego wyniki docelowo zasilają arkusze pod Invoicing Flow (dziś nieużywany). Szczegóły w sekcji „Zakres na dziś" na górze.

2a. Faza A — Zasilanie (soft księgowy → arkusz)

Apps Script w arkuszu woła endpointy */sheets-export na kontrolerach integracji. Handler (*ForSheetsQueryHandler / *AndAddToSheetCommandHandler) pobiera dane z softu, mapuje na uproszczone DTO i zapisuje je do arkusza przez OverrideRecordsInSheetByUrlAsync / AppendRecordsToSheetByUrlAsync (GoogleDriveService.cs). Część wzbogacania jest już w kodzie — np. handler Xero dokleja e-mail i telefon z zagnieżdżonego invoice.Contact.

2c. Faza C — Import (arkusz → flow → DB)

Azure Function z [TimerTrigger("* * * * *")] tika co minutę, ale to tylko puls schedulera (z distributed lock): sprawdza jedynie, które procesy są „due" wg ich harmonogramu (NextRunUtc, ScheduledSendHour, DaysInterval). Pojedynczy proces odświeża arkusz i wysyła najwyżej raz dziennie — o ustawionej godzinie, nie co minutę. Dopiero dla procesu „due" updater pobiera arkusz przez GoogleDriveService.DownloadDataFromPrivateGoogleSheetsAsync (autoryzowane Sheets API → konwersja na CSV), parsuje, mapuje i importuje do bazy.

  • Recurring: AutomatedRecurringCollectionRecurringCollectionFlowUpdater → ParseWithCustomFieldsAsync<RecurringInvoiceCsvModel> → wiersze bez DueDate są pomijane → import. Błąd konwersji CSV → alert mailowy (IAlertEmailService), nie twardy crash.
  • Invoicing (moduł dziś nieużywany — patrz „Zakres na dziś"): AutomatedInvoicingFlowUpdater (działa tylko gdy IntegrationType == GoogleSheets) → ParseToAsync<InvoiceSheetRow> → grupowanie po InvoiceGroupId → import draftów faktur.

2d. Faza D — Downstream

  • Recurring / Collection: zaimportowane dane napędzają automatyzację windykacji — komunikaty trafiają na Azure Service Bus (email-queue / sms-queue), skąd konsumenci (Azure Functions) wysyłają e-maile/SMS.
  • Invoicing: drafty → orchestracja Durable Functions → generowanie faktur w systemie docelowym (Xero / Fakturownia / inFakt).

3. Most: CustomFields

To jest mechanizm, dzięki któremu dowolne pola policzone w arkuszu przepływają do Sunbay bez zmian w kodzie — i dlatego arkusze są tak „elastyczne" (a migracja tak podstępna).

Application/Helpers/CsvParserService.cs → ParseWithCustomFieldsAsync<T>: dla każdego wiersza CsvHelper mapuje znane kolumny na właściwości modelu, a następnie ExtractCustomFields przechodzi po wszystkich nagłówkach CSV. Każdy nagłówek, który nie pasuje do żadnej publicznej właściwości modelu (porównanie po nazwie, case-insensitive, przez refleksję GetProperty(header, IgnoreCase|Public|Instance)), trafia do słownika Dictionary<string,string> CustomFields.

private static bool IsModelProperty<T>(string header) =>
    typeof(T).GetProperty(header,
        BindingFlags.IgnoreCase | BindingFlags.Public | BindingFlags.Instance) != null;
// nagłówek bez odpowiednika w modelu  ->  CustomFields[header] = value;
Ważny niuans

CustomFields obejmuje ścieżkę recurring/collection (RecurringInvoiceCsvModel, InvoicesCvsModel — implementują ICustomFields). Ścieżka invoicing używa ParseToAsync<InvoiceSheetRow> i nie przenosi dowolnych dodatkowych kolumn — liczą się tylko zdefiniowane pola.

4. Uwierzytelnienie do Google

Jedno konto serwisowe Google z domain-wide delegation; per klient stosowana jest impersonacja. Poświadczenia pochodzą z appsettings (klucze pod Google:Credentials; wartości sekretne z Key Vault — nie umieszczamy ich tutaj).

Ścieżka uwierzytelnienia do Sheets/Drive API
flowchart LR
  CFG["appsettings -> Google:Credentials (konto serwisowe Sunbay; szczegoly w Key Vault)"]
  GCF["GoogleClientFactory: GoogleCredential.FromJson()"]
  IMP["impersonacja per clientId (domain-wide delegation)"]
  SVC["SheetsService / DriveService (cache per email)"]
  GAPI["Google Sheets / Drive API"]
  CFG --> GCF --> IMP --> SVC --> GAPI

Odczyt arkusza ma dwie drogi w GoogleDriveService: publiczną (anonimowy export?format=csv) i prywatną (autoryzowane Spreadsheets.Values.Get + konwersja na CSV przez CsvHelper). Żywe flow używają drogi prywatnej (DownloadDataFromPrivateGoogleSheetsAsync).

DTO zasilające (faza A)

Każda integracja ma własne uproszczone DTO zapisywane do arkusza (np. SimplifiedXeroInvoiceDto z polami faktury + ContactEmail/ContactPhone). Kolejność kolumn w arkuszu = kolejność właściwości DTO (zapis przez refleksję, od komórki A2 — wiersz 1 to nagłówki). Pełne definicje: Application/Dto/SimplifiedModels/ oraz handlery w Application/Features/Invoicing/.

7. Granice migracji / co podmienić

Zależność procesów od arkuszy jest całkowita. Każdy żywy proces czyta dane z Google Sheets. W kodzie istnieje też IntegrationType.CsvUpload, ale służy on odrębnemu przypadkowi — importowi odbiorców (klientów naszych klientów) (frontend src/components/clients/ImportClients/ + src/components/csvImporterModal/; backend Sunbay.Invoices/Commands/ImportRecipients/, Queries/ExportRecipientsCsv/; źródło oznaczane InvoiceRecipientSource.Csv). To nie jest produkcyjna ścieżka zasilania procesów.

Szwy do podmiany przy migracji:

  • Odczyt: IGoogleDriveService.DownloadDataFromPrivateGoogleSheetsAsync — jeden punkt, w którym dane wchodzą do importu.
  • Zasilanie: handlery *ForSheets / *AndAddToSheet — dziś piszą do arkusza; docelowo źródło danych zamiast arkusza.
  • Transformacja: warstwa formuł/joinów z fazy B — dziś w arkuszu, do przeniesienia do realnego ETL (to największa, „niewidoczna" praca).
  • Model: IntegrationType — dodanie nowego źródła lub zastąpienie wartości GoogleSheets.
Pytania otwarte do ustalenia
  • Wolumeny: ile wierszy/faktur na arkusz, ile arkuszy/flow/klientów aktywnych?
  • Częstotliwość: jak często flow są „due" i jak często Apps Script odświeża zasilanie?
  • Które integracje księgowe są realnie używane dziś (a które tylko w kodzie)?
  • Jak wiele logiki z fazy B jest unikalne per klient vs. powtarzalne?

8. Legacy

Dla porządku: w kodzie istnieją też stare endpointy panelu komunikacji/dokumentów (api/2.0/sheets/*, [Obsolete] api/sheets/*) oraz zapis statusów/timestampów z powrotem do arkusza — są poza zakresem tej migracji i można je pominąć.
